Express is a framework that uses Node.js web
server functionality to simplify its API and add
other features. It makes it easier to organize applications
functionality with middleware and routing; it adds
utilities to Node.js HTTP objects; it facilitates
the rendering of dynamic HTML views; it defines an
easily implemented extensibility standard.

The javascript function that processes browser requests
in you application is called a request handler. In code,
its a function that takes to arguments: an object
that represents the requests and an object that
represents the response.

At a high level, Express provides four major features:

1. __Middleware__: Express has a middleware stack, which
is an array of functions. Rather than one monolithic request handler function,
you call several request handler functions that each
deal with a small chunck of work. These smaller requests
handler function are called middleware functions,
or middleware.

2. __Routing__: Functions that are called only when you
visit a specific URL with a specific method. These request handles are executed conditionally,
depending on what URL and HTTP method a client
sends.

3. __Extensions__ to requests and response objects:
extra methods and properties for developer convenience.

4. __Views__: Allow to dynamically render HTML. This
allows to change the HTML on the fly and to write
the HTML in other languages.

## 1. Middleware

Framework-free Node requires writing a single large
requests handler founction for your entire app. 
Middleware allows to break these request handles
functions into smaller bits. These smaller functions
tend to handle one thing at a time.

Rather than having just one handler, middleware
allows for many to happen in sequence. Rather than
having the request pass through one function, it
passes through an array of functions called a
middleware stack.

Each middleware function can modify the request or
the response, but it doesn't always have to. Eventually,
some middleware should respond to the request. If none
of them respond, then the server will hang and 
the browser will sit alone, without a response.

This is powerful because an application can be 
splitted into many small parts. These components
become easier to compose and reorder and it is 
also easy to pull third-party middleware.

### Middleware and the middleware stack

Web servers listen for request, parse those requests,
and send responses. The Node runtime will get these
requests firt and turn them from raw bytes into two
Javascript objects that you can hendel one object 
for the request (req) and one object for the response
(res).

In Express these objects are passed through an array
of functions, called the middleware stack. Express
will start at the first function in the stack and 
continue in order down the stack. Every function in 
this stack takes three arguments. The first twe are
req and res. They are given by Node. The third 
argument is itself a function, conventionally 
called `next`. When next is called, Express will go on
to the next function in the stack. Eventually, one 
of these functions must call `res.end`, which will
end the request.

There are two types of middleware. The first type
are regular middleware functions that take three
arguments. Most of the time, the app is in normal
mode, which looks only at theses middleware function
and skips the other.

The second kind is error-handling middleware. When 
the application is in error mode all regular 
middleware is ignored and Express will execute
only error-handling middleware functions. To enter
error mode, call next with an
argument. Its contion to call it with an error 
object.

These middleware functions take four arguments. The
first one is the error (the argument passed into 
`next`), and the remainder are: `req`, `res` and `next`.

Calling next with no arguments will exit error mode
and move onto the next normal middleware; calling
it with and argument will continue onto the next 
error-handling middleware if one exists.

### Other middleware

There is only one piece of middleware that is bundled
with Express: _express.static_. The Express team
maintains a number of middleware modules:
- body-parser (for parsign request bodies)
- cookie-parser (for parsing cookies from users,
paired with express-session to keep track of users)
- compression (for compressing responses to save on
bytes)

Other important middleware:
- Helmet (helps secure applications) <a href="https://helmetjs.github.io"><span class="oi oi-external-link" title="helmet" aria-hidden="true"></span></a>
- connect-asset (compiles and minifies CSS and Javasript assets)

Some third-party middleware:
- Morgan: logging middleware connect-rate limit
- Helmet: prevent some attacks
- cookie-parser: parses cookies
- response-time: sends the X-Response-time header to
debug the performance of application.

## 2. Routing

Routing is a way to map requests to specific handlers
on ther URL and HTTP verb.

The first argument is a path. The second argument is
a request handler function similar to middleware 
section. In addition to matching fixed routes, they
can match more complex ones.

Routing maps verbs and URIs to specific code. It maps
an HTTP verb + URI combo to a request handler.

Routes can have parameters. The simplest way to grab
a parameter is by putting it in the route with a colon
in front of it. To grab the value, you'll look inside
the params property of the request.

Express allows to sepecify routers as strings and to
specify them as regular expressions.

Another common way to dynamically pass information in
URLs is to use query strings. These can be retrieved
with `req.query`.

### Routers

A router is an isolated instance of middleware and
routes. Routers can be thought of as "mini" applications
only capable of performing middleware and routing.
Every Express application has a built-in app router.

Router behavve like middleware and can be ".use()d"
by the app in other routers. Routers allow to chunk
a big app into numerous miniapps that can be later put
together.

### Serving Static Files

`path.resolve` helps keep path resolution cross-plataform.

Middleware can be mounted at a given prefix.

{% highlight javascript %}
var apath = path.resolve(__dirname, "apath");
app.use("/otherpath", express.static("apath"));
{% endhighlight %}

Because express static is middleware, you can just
apply it multiple times.

## 4. Views

Views are the way to dynamically generate content(usually HTML).
A view engine is a module that does the actual rendering of views.

A number of different view engines are available to
dynamically generate HTML (EJS, Handlebars, Pug and others)

### View rendering

When the program calls `render`, what happens is this:

1. _Express builds up the context object every time you call render._
These context objects will get passed to the view engines when it's time
to render. These are effectively the variables available to views.

It first adds all the properties from `app.locals`, and object available to every request.
Then it adds all the properties in `res.locals`, overwriting anything added from `app.locals` if it was present.
Finally, it adds the properties of the object passed to `render`. 

2. _Express decide whether view caching is enabled._
Express caches the lookup of the view file and its assignment to the proper view engine.
It decides whether view caching is enabled in two ways:

   a. The option `app.enabled("view cache")`. By default, this is disabled in development
   mode and enabled in production, but it can be changed with `app.enable("view cache")` 
   or `app.disable(view cache)`.

   b. If the context object generated in the previous step has a truthy property called `cache`,
   then caching will be enabled for that view. This overrides any aplication setting. This enables to cache
   on a view-by-view basis.

3. _Express look up where the view file resides and what view engine to use._

4. _Express appends the default file extension if no extension was supplied._

5. _Express looks at the file extension to determine which engine to use._

6. _Express looks the file up in the `views` directory._ If it doen't find the file, it throws an error.

7. _Express caches all the lookup logic if it should._ If view caching enabled, it will cache all this lookup
logic for the next time.

8. _Express render the view_. This calls out the view engine and its one line in the Express's source code. This is where the view engine takes over and produces actual HTML.

### Making all view engines compatible with Express

Many of the view engines, such as EJS and Pug, will work with Express out of the box.
Others don't have an Express-compatible API and need to be wrapped in smoething Express can understand.

Consolidate.js (https://github.com/tj/consolidate.js) is a library that wraps a ton of view engines to
be compatible with Express. It has support for the classics like EJS, Pug, Mustache, Handlebars, and Hogan. To use it add to `app.js` 

{% highlight javascript %}
var engines = require('consolidate');
// and
app.engine(<engine_alias>, engines.<engine>);
{% endhighlight %}
